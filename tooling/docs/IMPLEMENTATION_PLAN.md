# Automated MDX Generation & Mintlify API Spec Sync — Implementation Plan

## Overview

This plan outlines the steps to build a fully automated system for generating MDX files and
enhancing Mintlify API documentation using LLM. The system bridges any API that produces an
OpenAPI spec and the Mintlify docs repo (`docs_v2`), keeping them in sync automatically.

The tooling lives in `docs_v2/tooling/` and is API-agnostic — it works with any OpenAPI spec,
not just the parser.

## Step 2: Develop the MDX File Generator

### 2.1 MDX Structure Requirements

Each MDX file must have:
- **Frontmatter**: `title`, `openapi` (links to spec operation), `description`
- **Overview section**: plain prose explaining the endpoint
- **`<ParamField>` blocks**: one per request parameter (auto-synced by `doc-sync`)
- **`<ResponseField>` blocks**: one per response field (auto-synced by `doc-sync`)
- **`<RequestExample>`** block: code samples in cURL, Python, JavaScript
- **`<ResponseExample>`** block: sample success/error responses
- **Configuration recipes** (optional): `<AccordionGroup>` with common use-case configs
- **Error handling section**: lists all error codes and their meanings

### 2.2 MDX Generator Script
- The existing `doc-sync` command handles *updating* existing MDX files.
- A new **stub generator** (`doc_sync/enrichment/stubs.py`) is needed to create MDX files
  from scratch for new endpoints.
- Input: OpenAPI operation JSON + endpoint metadata
- Output: fully formed `.mdx` file with all required sections

### 2.3 Customisation Parameters
The generator should accept:
- `--spec` - path to any `openapi.json` (v1, v2, or any other API)
- `--endpoint` - operation key (e.g. `POST /v2/parse/upload`)
- `--output` - destination MDX file path
- `--api` - optional label for the API (e.g. `parser-v1`, `parser-v2`)

### 2.4 Testing
- Run generator against both v1 and v2 spec files.
- Validate output MDX with `mintlify dev` to confirm it renders correctly.

## Step 3: Create the Makefile for Build Automation

Location: `parser/Makefile` (repo root)

### 3.1 Targets to Define

| Target | Description |
|---|---|
| `export-spec` | Run `cargo run --bin dump-openapi` -> writes `docs/openapi/v1.json` and `v2.json` |
| `sync-docs` | Run `doc-sync` against v1 spec for all MDX files |
| `sync-docs-v2` | Run `doc-sync` against v2 spec for v2 MDX files |
| `sync-docs-dry` | Dry-run sync (print diffs, no writes) |
| `generate-inventory` | Run `doc-inventory` -> writes `docs/api-documentation/parse-api-inventory.md` |
| `spec-coverage` | Run `doc-coverage` and report annotation coverage |
| `generate-mdx-stubs` | Generate new MDX stub files for any undocumented endpoints |

### 3.2 Example Makefile

```makefile
DOCS_ROOT   := ../docs_v2
TOOLING_DIR := $(DOCS_ROOT)/tooling
SPEC_V1     := docs/openapi/v1.json
SPEC_V2     := docs/openapi/v2.json

.PHONY: export-spec sync-docs sync-docs-v2 sync-docs-dry \
        generate-inventory spec-coverage generate-mdx-stubs

export-spec:
	cd core && cargo run --bin dump-openapi

sync-docs: export-spec
	uv --project $(TOOLING_DIR) run doc-sync \
	  --spec $(SPEC_V1) --docs-root $(DOCS_ROOT)

sync-docs-v2: export-spec
	uv --project $(TOOLING_DIR) run doc-sync \
	  --spec $(SPEC_V2) --docs-root $(DOCS_ROOT)

sync-docs-dry: export-spec
	uv --project $(TOOLING_DIR) run doc-sync \
	  --spec $(SPEC_V1) --docs-root $(DOCS_ROOT) --dry-run

generate-inventory: export-spec
	uv --project $(TOOLING_DIR) run doc-inventory \
	  --spec $(SPEC_V1) --output docs/api-documentation/parse-api-inventory.md

spec-coverage: export-spec
	uv --project $(TOOLING_DIR) run doc-coverage \
	  --spec $(SPEC_V1) --verbose

generate-mdx-stubs: export-spec
	uv --project $(TOOLING_DIR) run doc-sync \
	  --spec $(SPEC_V2) --docs-root $(DOCS_ROOT)
```

## Step 4: Develop the Async Pipeline for API Spec Generation

### 4.1 Spec Fetch / Export
- Source of truth: `core/src/` Rust types with `#[utoipa::path]` and `#[derive(ToSchema)]` annotations.
- Export: `cargo run --bin dump-openapi` writes `docs/openapi/v1.json` and `v2.json`.
- Both specs are self-contained (inline all `$ref` schemas that matter).
- For other APIs, any tool that produces a valid `openapi.json` can feed into `doc-sync`.

### 4.2 LLM Enrichment
- `doc_sync/enrichment/enrich.py` sends existing MDX + spec operation JSON to Claude.
- Claude updates only `<ParamField>` / `<ResponseField>` blocks.
- All hand-written prose, code examples, and accordion configs are preserved.
- Model: `claude-sonnet-4-6` (Anthropic API) or `us.anthropic.claude-opus-4-6-v1` (Bedrock) — configurable via `--model` flag.
- System prompt is in `doc_sync/prompts/enrich_endpoint.md` — edit it to tune output quality.
- Add `--bedrock` to any command to route via Amazon Bedrock instead of the Anthropic API.

### 4.3 Async Execution
- For large docs sets, `doc_sync/cli/sync.py` can be extended to process files concurrently
  using `asyncio` + the `anthropic` async client (`AsyncAnthropic`).
- Recommended: process files in batches of 5 to respect API rate limits.

### 4.4 Output Storage
- Updated MDX files are written back to `docs_v2/api-reference/<api-name>/`.
- Diffs are printed to stdout for review before committing.

## Step 5: Compare and Merge the API Specs

### 5.1 Diff Workflow
- `doc-sync --dry-run` shows a unified diff per file without writing.
- `doc_sync/mdx/writer.py` uses `difflib` to compute and display coloured diffs via `rich`.

### 5.2 Identifying LLM Enhancements
- Compare `<ParamField>` descriptions before/after to verify LLM enrichment quality.
- `doc-coverage` reports percentage of schema properties with descriptions.

### 5.3 Merge Strategy
- Run `doc-sync` on a feature branch in the docs repo.
- Open a PR for human review before merging to main.
- CI can run `doc-sync --dry-run` to flag drift between spec and docs.

### 5.4 Readability Review
- After merge, run `mintlify dev` to preview rendered output.
- Validate that all `<ParamField>` types match the OpenAPI spec types.
